# Reading ELLA Rebuild v1 PRD

## 1. 제품 정의

제품 이름:
Reading ELLA

이 제품은 영어 리딩 문제를 파일 기반으로 관리하고,
학생이 웹에서 풀이할 수 있게 하는 경량 학습 도구다.

핵심 구조는 아래와 같다.

- 문제 원본: JSON
- 번역 오버레이: JSON
- 백엔드: FastAPI
- 학습 기록: SQLite
- 프론트엔드: Next.js

## 2. 문제 정의

기존 빠른 프로토타입은 검증에는 유리했지만,
앞으로 계속 확장하려면 화면 기술과 운영 구조를 다시 정리할 필요가 있다.

이번 리빌드에서 해결하려는 문제는 아래와 같다.

- 문제를 계속 수동으로 생산하고 관리하기 어렵다
- 화면을 더 일반적인 웹 프론트엔드 구조로 바꾸고 싶다
- 채점, 오답정리, 팔로업 처리를 남길 저장 구조가 필요하다
- 학생이 틀린 이유를 쉽게 다시 이해할 장치가 필요하다
- TTS와 번역 같은 보조 학습 기능이 필요하다
- 하지만 이 보조 기능이 항상 비용을 발생시키면 안 된다
- 외부 API 비용을 기본적으로 최소화하며 운영해야 한다
- 데이터 구조는 단순해야 한다

## 3. 목표 사용자

### 운영자

- Python으로 문제를 만드는 사람
- 문제를 검수하고 반영하는 사람

### 학생

- 웹에서 리딩 문제를 푸는 사용자

## 4. 핵심 목표

### 운영자 목표

- 새 pack을 빠르게 만들 수 있다
- pack 형식을 검사할 수 있다
- 여러 pack을 파일로 관리할 수 있다

### 학생 목표

- 웹에서 바로 문제를 시작할 수 있다
- 6문항을 풀 수 있다
- 제출 후 결과를 바로 볼 수 있다
- 필요할 때 지문과 문항을 들을 수 있다
- 필요할 때 번역을 선택적으로 볼 수 있다
- 필요할 때 오답 해석을 선택적으로 볼 수 있다

### 시스템 목표

- 기본 운영비를 거의 0원으로 유지한다
- 문제 원본은 JSON 파일로 관리한다
- 번역 자산도 JSON 파일로 관리할 수 있다
- 학생 기록은 SQLite로 저장한다
- 실시간 외부 API 호출은 기본 꺼짐 상태를 유지한다

## 6. 사용자 시나리오

### 운영자 시나리오

1. Python 스크립트로 새 문제 템플릿을 만든다.
2. 지문과 6문항을 수동으로 작성한다.
3. 검증 스크립트를 돌린다.
4. 통과한 문제를 `packs/`에 둔다.
5. 프론트에서 바로 확인한다.

### 학생 시나리오

1. 웹 화면에 들어간다.
2. 난이도를 선택한다.
3. 문제를 읽고 답을 고른다.
4. 제출한다.
5. 점수와 오답을 확인한다.
6. 필요하면 틀린 문항을 번역하거나 오답 해석을 다시 본다.
7. 필요하면 지문이나 보기 TTS를 다시 듣는다.

## 7. 기능 요구사항

### A. 문제 로드

- `packs/*.json`에서 문제를 가져온다
- `pack_id` 또는 난이도 기준 선택이 가능해야 한다
- 형식이 잘못된 pack은 사용자에게 오류로 보여야 한다

### B. 문제 풀이

- 지문 1개를 보여준다
- 문제는 6개 고정이다
- 각 문항은 4지선다다
- 답은 화면 상태에 저장한다

### C. 제출

- 6문항을 모두 고르기 전에는 제출되지 않아야 한다
- 제출 후에는 총점과 스킬별 점수를 보여준다

### D. 결과 저장

- 제출 결과는 SQLite에 저장한다
- 시도 기록, 문항별 정오답, 스킬별 점수가 저장되어야 한다
- 틀린 문항은 팔로업용으로 다시 조회할 수 있어야 한다

### E. 보조 학습 기능

- 지문 TTS를 켜고 끌 수 있어야 한다
- 문항 TTS를 켜고 끌 수 있어야 한다
- 보기 TTS를 켜고 끌 수 있어야 한다
- 지문 번역을 선택적으로 열 수 있어야 한다
- 문항 번역을 문항 단위로 선택적으로 열 수 있어야 한다
- 보기 번역을 문항 단위로 선택적으로 열 수 있어야 한다
- 오답 해석을 짧게 또는 자세히 선택적으로 요청할 수 있어야 한다
- 외부 API 사용은 프론트엔드 토글을 통해 사용자가 직접 켜야 한다
- 외부 API 토글이 꺼져 있으면 로컬 자산만 사용해야 한다

## 8. 비기능 요구사항

- 빠르게 로딩되어야 한다
- 구조가 단순해야 한다
- 사람이 파일만 봐도 pack을 이해 가능해야 한다
- 비용이 거의 들지 않아야 한다
- 보조 학습 기능은 기본적으로 비용 없는 방식부터 우선 사용해야 한다

## 9. 비용 정책

이번 프로젝트의 중요한 조건은 "비용 최소화"다.

따라서 아래를 기본 원칙으로 둔다.

- 실시간 외부 유료 API의 자동 호출 금지
- 문제 원본은 로컬 JSON 기반
- 번역 자산은 로컬 JSON 기반 우선
- 학습 기록은 로컬 SQLite 기반
- 문제 생성도 기본은 수동
- 브라우저 TTS 같은 무상 기능을 우선 사용
- 외부 API는 사용자나 운영자가 명시적으로 켠 경우만 사용

텍스트 보조 기능에서 쓰는 모델은 아래 2개를 기준으로 한다.

- `gemini-3.1-flash-lite-preview`: 기본 번역, 짧은 오답 해설
- `gemini-2.5-flash`: 더 긴 오답 해석, 추론형 문항 설명, 팔로업 설명

외부 API는 아래 경우만 예외적으로 허용한다.

- 운영자가 문제 초안을 수동으로 만들 때
- 운영자가 번역 초안을 수동으로 만들 때
- 학생이 번역, 오답 해석, 또는 고품질 TTS를 직접 요청했을 때

단, 학생용 외부 API 사용은 아래 조건을 만족해야 한다.

- 프론트엔드에서 토글로 직접 켠 경우만 허용
- 기본값은 OFF
- 어떤 기능이 외부 API를 쓰는지 화면에서 보여야 한다
- API 키는 프론트에 두지 않고 백엔드에서 관리한다

단, 이 경우에도 자동 배포는 금지하고 사람이 최종 검수해야 한다.

## 10. 성공 기준

- pack 파일을 수동으로 계속 추가할 수 있다
- FastAPI가 문제를 정상 제공하고 채점한다
- Next.js 화면이 문제를 정상 표시한다
- 학생이 6문항을 풀고 제출할 수 있다
- 점수가 정확하다
- 오답과 팔로업 데이터가 SQLite에 저장된다
- 번역, TTS, 오답 해석이 선택 기능으로 동작한다
- API 토글이 OFF일 때는 외부 API 호출이 없다
- 프론트엔드 실행과 빌드는 `pnpm` 기준으로 정리되어 있다
