# Rebuild v1 Content Model

## 1. 문서 목적

이 문서는 "콘텐츠를 무엇으로 구성할 것인가"를 정하는 문서다.
UX보다 먼저 결정해야 하는 기준 문서다.

쉽게 말하면:
학생이 실제로 보게 될 학습 재료의 설계도다.

## 2. 핵심 원칙

이번 리빌드의 콘텐츠는 3개 층으로 나눈다.

1. 기본 문제 콘텐츠
2. 보조 학습 콘텐츠
3. 학습 결과로 생기는 복습 콘텐츠

즉:
문제 원본, 번역/TTS 같은 도움 요소, 오답/팔로업 데이터를 분리해서 본다.

## 3. 기본 문제 콘텐츠

## 3.1 기본 단위

1개의 기본 학습 단위는 아래로 정의한다.

```text
1 pack = 지문 1개 + 객관식 6문항
```

이 구조를 v1의 기본 단위로 고정한다.

## 3.2 문항 구성

문항 수는 6개 고정이다.

권장 스킬 분포:

- `main_idea`: 1
- `detail`: 2
- `inference`: 1
- `vocab_in_context`: 2

이 분포를 권장 기준으로 둔다.

## 3.3 한 문항의 기본 구성

한 문항은 아래 요소를 가진다.

- 문항 id
- 스킬
- 질문 문장
- 보기 4개
- 정답 번호
- 해설 또는 rationale

## 3.4 운영자가 직접 만드는 것

운영자는 기본적으로 아래를 수동으로 만든다.

- 지문 제목
- 지문 본문
- 6문항
- 보기 4개씩
- 정답
- 해설

즉, 영어 원본 문제는 수동 제작이 기본이다.

## 4. 보조 학습 콘텐츠

보조 학습 콘텐츠는 학생 이해를 돕는 선택 기능이다.

v1에서 필요한 보조 학습 콘텐츠는 아래 3종이다.

- 번역
- TTS
- 오답 해석

## 4.1 번역 모델

번역은 원본 문제와 분리된 오버레이로 관리한다.

번역 단위는 아래처럼 나눈다.

- 지문 제목 번역
- 지문 본문 번역
- 문항 prompt 번역
- 보기 번역

즉, 번역은 "문항 전체"가 아니라 더 잘게 나뉜다.
그래야 학생이 필요한 부분만 볼 수 있다.

## 4.2 번역 표시 방식

학생은 번역을 항상 보지 않는다.
필요할 때만 선택적으로 연다.

권장 표시 단위:

- 지문 번역 열기
- 문항 번역 열기
- 보기 번역 열기

## 4.3 번역 출처 우선순위

번역은 아래 우선순위로 사용한다.

1. 수동 번역 JSON
2. 캐시된 API 번역
3. 사용자가 직접 요청한 실시간 API 번역

이 순서를 쓰는 이유는 비용과 품질 때문이다.

- 수동 번역이 가장 안정적이다
- 캐시된 번역은 비용을 다시 줄인다
- 실시간 API 번역은 마지막 선택지다

권장 모델 역할:

- `gemini-3.1-flash-lite-preview`: v1 기본 번역 전체

## 4.4 TTS 모델

TTS는 텍스트를 읽어주는 기능이다.

TTS 지원 단위:

- 지문 읽기
- 문항 읽기
- 보기 읽기

## 4.5 TTS 우선순위

TTS는 아래 순서를 권장한다.

1. 브라우저 기본 TTS
2. 사용자가 요청한 외부 TTS API

즉, 기본은 비용 없는 방식으로 시작한다.

## 4.6 TTS 사용 방식

학생은 아래 버튼을 선택적으로 누를 수 있어야 한다.

- 지문 읽기
- 문항 읽기
- 보기 읽기
- 정지

항상 자동으로 읽기 시작하면 안 된다.

## 4.7 오답 해석 모델

오답 해석은 학생이 제출 후에만 보는 선택 기능이다.

오답 해석 출처 우선순위:

1. pack 안의 `rationale`
2. 캐시된 해설
3. 사용자가 직접 요청한 실시간 API 해설

권장 모델 역할:

- `gemini-3.1-flash-lite-preview`: 한 줄 힌트, 짧은 설명
- `gemini-2.5-flash`: 자세한 오답 분석, 추론형 문제 해설, 팔로업 설명

## 5. API 토글 모델

## 5.1 왜 필요한가

이번 프로젝트는 외부 API를 완전히 금지하는 것이 아니라,
"필요할 때만 직접 켜서 쓰는 구조"를 목표로 한다.

그래서 프론트엔드에는 API 사용 토글이 필요하다.

## 5.2 기본 규칙

- API 토글 기본값은 OFF
- 사용자가 직접 켜야 한다
- 어떤 기능이 API를 쓰는지 화면에서 보여야 한다
- API 키는 프론트에 저장하지 않는다

## 5.3 권장 토글 종류

- `Use Assist API`
- `Use API Translation`
- `Use API TTS`
- `Use API Explain`

아주 단순하게 가려면 첫 단계에서는 `Use Assist API` 하나만 둬도 된다.

## 5.4 토글이 OFF일 때

- 수동 번역 JSON만 사용
- 브라우저 TTS만 사용
- 로컬 `rationale`만 사용
- 외부 API는 호출하지 않음

## 5.5 토글이 ON일 때

- 번역 자산이 없으면 외부 번역 API를 요청할 수 있음
- 더 좋은 음성이 필요하면 외부 TTS API를 요청할 수 있음
- 더 자세한 오답 해석이 필요하면 외부 설명 API를 요청할 수 있음

단, 이것도 자동 실행이 아니라 버튼 기반 요청이어야 한다.

## 6. 학습 결과로 생기는 복습 콘텐츠

학생이 문제를 풀면 새로운 콘텐츠가 생긴다.

예:

- 틀린 문제 목록
- 약한 스킬 목록
- 다시 풀어야 할 문제 목록

이건 운영자가 만드는 콘텐츠가 아니라 시스템이 만드는 복습 콘텐츠다.

## 6.1 최소 복습 단위

v1에서는 아래 정도면 충분하다.

- 틀린 문항 1개
- 그 문항의 스킬
- 내가 고른 답
- 정답
- 다시 보기 상태
- 짧은 오답 해석 또는 자세한 오답 해석

## 6.2 팔로업 규칙

v1에서는 단순하게 시작한다.

- 틀린 문제는 follow-up 대상에 넣는다
- 약한 스킬은 결과 화면에서만 먼저 보여준다
- 오답 해석은 결과 화면에서 바로 보거나 나중에 다시 볼 수 있다

즉, 첫 단계에서는 "틀린 문제 다시 보기"만 잘 되면 충분하다.

## 7. 콘텐츠 제작 흐름

운영자 기준 콘텐츠 흐름은 아래와 같다.

1. 레벨(`GT / S / MGT`)을 선택해 Python 스크립트로 pack 뼈대 생성
2. 문제 수동 작성
3. pack 검증
4. 필요하면 번역 템플릿 생성
5. 번역 수동 작성 또는 API 보조 생성
6. 번역 검증
7. 배포

## 8. UX에 미치는 영향

이 콘텐츠 모델이 정해지면 UX는 자연스럽게 아래처럼 나온다.

- 시작 화면: 난이도 / 시작 / API 토글
- 문제 화면: 지문 + 6문항 + 번역 버튼 + TTS 버튼
- 결과 화면: 점수 + 오답 + 약한 스킬 + 오답 해석 버튼
- 복습 화면: 틀린 문제 다시 보기

즉, UX는 콘텐츠 구조에서 파생된다.

## 9. 이번 문서의 결론

이번 프로젝트의 콘텐츠는 아래 3층으로 구성한다.

1. 영어 원본 문제 pack
2. 선택형 번역 / TTS 보조층
3. 오답 / 팔로업 복습층

이 구조를 먼저 고정하면,
그 다음에 UX와 API 설계를 훨씬 안정적으로 할 수 있다.
