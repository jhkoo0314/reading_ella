# Rebuild v1 Screen Spec

## 1. 문서 목적

이 문서는 각 화면에 무엇이 보여야 하는지 정하는 문서다.

쉽게 말하면:
화면별 체크리스트다.

`08_UX_FLOW.md`가 "어떤 순서로 이동하는가"를 정하는 문서라면,
이 문서는 "각 화면 안에 무엇이 있어야 하는가"를 정하는 문서다.

## 2. 화면 목록

v1에서 정의하는 핵심 화면은 아래 4개다.

1. 시작 화면
2. 문제 풀이 화면
3. 결과 화면
4. 복습 화면

추가로 작은 보조 화면 또는 레이어가 있다.

- 제출 확인 모달
- 번역 펼침 영역
- TTS 제어 영역
- 오답 해석 영역
- 오류 / 빈 상태 메시지

## 3. 공통 화면 규칙

모든 화면에 공통으로 적용할 규칙이다.

- 버튼 문구는 짧고 분명해야 한다
- 기본 상태에서 번역은 닫혀 있어야 한다
- 기본 상태에서 TTS는 재생되지 않아야 한다
- 기본 상태에서 외부 API 토글은 OFF여야 한다
- 외부 API를 쓰는 버튼은 화면에서 구분되어야 한다
- 제출 전에는 정답과 해설이 보이면 안 된다
- 학생이 지금 어디 있는지 쉽게 알아야 한다

## 4. 시작 화면

## 4.1 화면 목적

학생이 가장 빨리 학습을 시작하게 하는 화면이다.

## 4.2 필수 요소

- 서비스 제목
- 한 줄 설명
- 난이도 선택 영역
- `문제 시작` 버튼
- `오답 복습` 버튼
- API 사용 토글 카드

## 4.3 권장 문구

제목 예시:

`Reading ELLA`

설명 예시:

`지문 1개와 6문항을 풀고 바로 결과를 확인할 수 있습니다.`

## 4.4 난이도 선택 영역

필수 항목:

- `GT`
- `S`
- `MGT`

표시 방식:

- 버튼 3개
- 또는 카드 3개

선택된 난이도는 시각적으로 분명히 보여야 한다.

## 4.5 API 사용 토글 카드

카드 안에는 아래가 보여야 한다.

- `Use Assist API`
- `Use API Translation`
- `Use API TTS`
- `Use API Explain`
- 현재 상태 표시: `OFF` 또는 `ON`

첫 버전에서는 필요하면 `Use Assist API` 하나만 먼저 둘 수 있다.

## 4.6 시작 화면 버튼 규칙

`문제 시작` 버튼:

- 난이도를 고르면 활성화
- 난이도를 안 고르면 비활성화

`오답 복습` 버튼:

- 복습 데이터가 없으면 비활성화 가능
- 또는 눌렀을 때 빈 상태 화면을 보여줄 수 있음

## 4.7 시작 화면에서 보이면 안 되는 것

- 문제 본문 미리보기
- 정답 수
- 이전 문제의 정답
- 자동 팝업 해설

## 5. 문제 풀이 화면

## 5.1 화면 목적

학생이 한 세트 문제를 풀게 하는 메인 화면이다.

## 5.2 화면 레이아웃

권장 구조:

```text
Header
Passage Section
Question List
Sticky Submit Area
```

## 5.3 Header

필수 요소:

- 현재 난이도
- pack id 또는 세트 이름
- 미응답 문항 수
- `제출` 버튼
- `홈으로` 버튼 또는 뒤로 이동 수단

## 5.4 Passage Section

필수 요소:

- 지문 제목
- 지문 본문
- 지문 번역 열기 버튼
- 지문 듣기 버튼
- 지문 정지 버튼

선택 요소:

- 단어 수
- topic 표시

## 5.5 Passage Section 동작 규칙

- 번역은 버튼을 눌렀을 때만 펼쳐진다
- 로컬 번역이 있으면 바로 보여준다
- 로컬 번역이 없고 API 토글이 OFF면 번역 없음 문구를 보여준다
- 지문 듣기는 버튼을 눌렀을 때만 시작한다
- 다른 TTS가 재생 중이면 이전 재생은 멈춘다

## 5.6 Question Card

각 문항 카드에 필요한 요소:

- 문항 번호
- 스킬 배지
- 질문 문장
- 선택지 4개
- 현재 선택 표시
- 문항 번역 열기 버튼
- 보기 번역 열기 버튼
- 문항 듣기 버튼
- 보기 듣기 버튼

## 5.7 Choice UI 규칙

- 한 문항에는 하나의 답만 선택 가능
- 선택 즉시 강조 상태를 보여준다
- 선택 후에도 다시 바꿀 수 있어야 한다
- 선택 전에는 아무 것도 미리 선택하지 않는다

## 5.8 문제 화면 하단

필수 요소:

- 현재 미응답 개수
- `제출` 버튼

권장 요소:

- `모든 문항 보기`
- `위로 이동`

모바일에서는 하단 제출 영역을 고정하는 것이 좋다.

## 5.9 문제 화면 금지 사항

- 자동 제출
- 자동 번역 열림
- 자동 TTS 시작
- 정답 노출
- rationale 노출

## 6. 제출 확인 모달

## 6.1 목적

실수 제출을 막기 위한 짧은 확인 창이다.

## 6.2 필수 요소

- 제목
- 남은 미응답 수 또는 제출 확인 문구
- `취소`
- `제출하기`

## 6.3 문구 예시

모든 문항 응답 완료 시:

`제출 후 바로 결과가 표시됩니다. 제출할까요?`

미응답이 있을 때:

`아직 답하지 않은 문항이 있습니다. 모두 답한 뒤 제출해주세요.`

## 7. 결과 화면

## 7.1 화면 목적

학생이 바로 점수와 틀린 이유를 이해하는 화면이다.

## 7.2 상단 요약 카드

필수 요소:

- 총점
- 맞은 개수 / 전체 개수
- 간단한 한 줄 결과 문구

문구 예시:

`6문항 중 4문항을 맞혔습니다.`

## 7.3 스킬별 결과 카드

필수 요소:

- `main_idea`
- `detail`
- `inference`
- `vocab_in_context`

각 카드에는 아래를 표시한다.

- 맞은 개수
- 전체 개수

## 7.4 틀린 문항 리스트

각 틀린 문항 카드에 필요한 것:

- 문항 번호
- 스킬
- 내가 고른 답
- 정답
- 짧은 오답 해석 버튼
- 자세한 오답 해석 버튼
- 문항 번역 열기 버튼
- 보기 번역 열기 버튼

## 7.5 오답 해석 영역

오답 해석을 펼쳤을 때 필요한 것:

- 해석 본문
- 해석 출처 표시
- 사용 모델 표시
- 닫기 버튼

출처 예시:

- `local rationale`
- `cached explanation`
- `api live`

모델 예시:

- `gemini-3.1-flash-lite-preview`
- `gemini-2.5-flash`

## 7.6 결과 화면 주요 버튼

- `짧게 보기`
- `자세히 보기`
- `다시 복습하기`
- `홈으로`

## 7.7 결과 화면 금지 사항

- 맞은 문제까지 모두 길게 펼쳐서 복잡하게 만드는 것
- 외부 API 자동 호출
- 학생이 요청하지 않은 상세 해설 자동 생성

## 8. 복습 화면

## 8.1 화면 목적

틀린 문제만 다시 확인하는 화면이다.

## 8.2 필수 요소

- 화면 제목
- 틀린 문항 목록
- 스킬 표시
- 내가 고른 답
- 정답
- 오답 해석 다시 보기 버튼
- 번역 보기 버튼
- TTS 버튼
- `홈으로` 버튼

## 8.3 복습 화면 규칙

- v1에서는 틀린 문제만 보여준다
- 맞은 문제는 기본으로 보여주지 않는다
- 복습 화면에서도 외부 API는 자동 호출하지 않는다

## 9. 보조 UI 컴포넌트 명세

## 9.1 Translation Panel

필수 요소:

- 제목 또는 라벨
- 번역 본문
- 닫기 버튼
- 번역 출처 표시

## 9.2 TTS Controls

필수 요소:

- 재생 버튼
- 정지 버튼

권장 요소:

- 현재 재생 중 표시

## 9.3 API Status Badge

필수 요소:

- `API OFF`
- `API ON`

이 배지는 시작 화면과 결과 화면에 보이는 것이 좋다.

## 9.4 Empty State Card

필수 요소:

- 상태 문구
- 돌아가기 버튼 또는 닫기 버튼

## 10. 상태별 표시 규칙

## 10.1 Loading

표시 예:

- `문제를 불러오는 중입니다.`
- `결과를 계산하는 중입니다.`
- `오답 해석을 불러오는 중입니다.`

## 10.2 Empty

표시 예:

- `현재 복습할 오답이 없습니다.`
- `이 항목은 아직 번역이 준비되지 않았습니다.`

## 10.3 Error

표시 예:

- `문제를 불러오지 못했습니다.`
- `결과 저장에 실패했습니다.`
- `추가 해설을 불러오지 못했습니다.`

## 10.4 Success

표시 예:

- `결과가 저장되었습니다.`
- `오답 해석이 준비되었습니다.`

## 11. 반응형 규칙

모바일과 데스크톱 모두 고려해야 한다.

### 모바일

- 한 줄에 너무 많은 정보 배치 금지
- 제출 버튼은 하단 고정 권장
- 번역과 해설은 아코디언 방식 권장

### 데스크톱

- 지문과 문항을 한 화면에서 충분히 읽을 수 있어야 한다
- 결과 화면에서 요약 카드와 오답 리스트를 구분해 보여주는 것이 좋다

## 12. 접근성 기본 규칙

- 버튼 라벨은 의미가 분명해야 한다
- 선택된 답은 색상 외 방식으로도 보여야 한다
- TTS 버튼과 정지 버튼은 분리한다
- 번역 열기와 해설 열기 버튼은 역할이 다르게 보이게 한다

## 13. v1 디자인 복잡도 기준

v1 화면은 아래를 지킨다.

- 정보 구조는 단순하게
- 한 화면에 너무 많은 카드 금지
- AI 기능은 보조 역할만
- 학생의 기본 행동은 항상 `문제 풀이`가 중심

즉, 화면은 똑똑해 보여야 하는 것이 아니라,
학생이 덜 헷갈리게 만들어야 한다.

## 14. 디자인 구현 역할 분담

이 문서는 화면 구성 기준을 정하는 문서지만,
실제 구현 담당도 아래처럼 나눈다.

- 코덱스 CLI: 화면 구조, 컴포넌트 뼈대, 상태 연결, API 연결, 기본 동작 UI
- 안티그래비티: 본격적인 프론트엔드 디자인, 비주얼 완성도, 최종 스타일 구현

중요:

- 코덱스 CLI는 프론트엔드 디자인 구현에서 먼저 뼈대만 구현한다
- 최종적인 디자인 표현과 화면 완성도는 안티그래비티에서 맡는다
