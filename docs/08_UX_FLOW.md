# Rebuild v1 UX Flow

## 1. 문서 목적

이 문서는 학생이 화면을 어떤 순서로 쓰는지 정하는 문서다.

쉽게 말하면:
버튼을 어디서 누르고, 다음에 어디로 가는지 정하는 문서다.

이 문서는 디자인 시안보다 먼저 읽는 기준 문서다.

## 2. UX 핵심 원칙

- 처음 보는 학생도 바로 시작할 수 있어야 한다
- 문제 풀이 흐름은 짧고 단순해야 한다
- 번역, TTS, 오답 해석은 모두 "선택 기능"이어야 한다
- 외부 API는 자동 실행되면 안 된다
- 제출 전에는 정답이나 해설이 보이면 안 된다
- 제출 후에는 오답 복습이 빠르게 이어져야 한다

## 3. 1차에서 다루는 사용자

v1에서 UX를 먼저 설계할 대상은 학생 1명이다.

즉:

- 로그인 없음
- 다중 사용자 관리 없음
- 학생용 웹 흐름이 가장 중요함

운영자용 문제 제작 흐름은 별도 스크립트 문서로 관리한다.

## 4. 전체 화면 흐름

학생용 기본 흐름은 아래 4단계다.

```text
시작 화면
  -> 문제 풀이 화면
  -> 결과 화면
  -> 복습 화면
```

더 자세히 쓰면 아래와 같다.

```text
Home
  -> Start Session
  -> Solve Pack
  -> Submit
  -> Result
  -> Review Wrong Answers
  -> Return Home
```

v1에서는 이 흐름을 기본으로 고정한다.

## 5. 시작 화면

## 5.1 화면 목적

학생이 바로 문제를 시작할 수 있게 하는 화면이다.

## 5.2 꼭 보여야 하는 것

- 서비스 이름 또는 간단한 제목
- 난이도 선택
- `문제 시작` 버튼
- `오답 복습` 버튼
- API 사용 토글 영역

## 5.3 API 토글 영역

기본값은 모두 OFF로 시작한다.

권장 항목:

- `Use Assist API`
- `Use API Translation`
- `Use API TTS`
- `Use API Explain`

아주 단순하게 가려면 첫 단계는 `Use Assist API` 하나만 둬도 된다.

## 5.4 학생이 여기서 결정하는 것

- 오늘 풀 문제 난이도
- 보조 API를 쓸지 말지

단, 이 화면에서 너무 많은 설정을 먼저 묻지 않는다.

이유:

- 시작 전에 피로감을 주기 쉽기 때문이다
- 기본 목표는 빨리 시작하는 것이기 때문이다

## 6. 문제 풀이 화면

## 6.1 화면 목적

지문 1개와 6문항을 풀게 하는 핵심 화면이다.

## 6.2 화면 구성

권장 구조는 아래와 같다.

```text
상단 바
  - 난이도
  - 남은 미응답 수
  - 제출 버튼

본문
  - 지문 영역
  - 지문 번역 버튼
  - 지문 TTS 버튼

문항 목록
  - 문항 1
  - 문항 2
  - 문항 3
  - 문항 4
  - 문항 5
  - 문항 6
```

## 6.3 지문 영역

지문 영역에는 아래가 있어야 한다.

- 제목
- 본문
- 지문 번역 열기 버튼
- 지문 읽기 버튼
- 지문 읽기 정지 버튼

번역은 기본으로 닫혀 있어야 한다.

## 6.4 문항 영역

각 문항 카드에는 아래가 있어야 한다.

- 문항 번호
- 스킬 표시
- 질문 문장
- 보기 4개
- 문항 번역 열기 버튼
- 보기 번역 열기 버튼
- 문항 읽기 버튼
- 보기 읽기 버튼

학생은 답을 고를 때 즉시 선택 상태를 볼 수 있어야 한다.

## 6.5 제출 규칙

- 6문항을 모두 고르기 전에는 제출을 막는다
- 제출 버튼을 누르면 마지막 확인을 한 번 보여준다
- 확인 후에는 채점을 진행한다

권장 확인 문구:

`제출 후 바로 결과가 표시됩니다. 제출할까요?`

## 6.6 문제 화면에서 금지할 것

- 자동 번역 열기
- 자동 TTS 재생
- 자동 외부 API 호출
- 제출 전 정답 공개
- 제출 전 해설 공개

## 7. 결과 화면

## 7.1 화면 목적

학생이 방금 푼 결과를 바로 이해하게 하는 화면이다.

## 7.2 꼭 보여야 하는 것

- 총점
- 맞은 개수 / 전체 개수
- 스킬별 결과
- 틀린 문항 목록
- `다시 복습하기` 버튼
- `홈으로` 버튼

## 7.3 틀린 문항 카드

틀린 문항 카드에는 아래 정보가 필요하다.

- 문항 번호
- 내가 고른 답
- 정답
- 스킬
- 짧은 오답 해석 버튼
- 자세한 오답 해석 버튼
- 문항 번역 열기 버튼
- 보기 번역 열기 버튼

## 7.4 오답 해석 규칙

오답 해석은 제출 후에만 볼 수 있다.

우선순위:

1. pack 안의 `rationale`
2. 저장된 해설
3. API 기반 추가 해설

모델 권장 기준:

- 짧은 해석: `gemini-3.1-flash-lite-preview`
- 자세한 해석: `gemini-2.5-flash`

## 7.5 결과 화면에서 학생이 해야 하는 행동

학생은 결과 화면에서 아래 중 하나를 고른다.

- 틀린 문제를 바로 읽어본다
- 짧은 해설만 확인한다
- 자세한 해설을 요청한다
- 복습 화면으로 이동한다
- 홈으로 돌아간다

## 8. 복습 화면

## 8.1 화면 목적

틀린 문제만 다시 보는 화면이다.

v1에서는 복습 화면을 너무 복잡하게 만들지 않는다.

## 8.2 꼭 보여야 하는 것

- 틀린 문항 목록
- 각 문항의 스킬
- 이전에 고른 답
- 정답
- 오답 해석 다시 보기
- 홈으로 돌아가기 버튼

## 8.3 v1 복습 방식

v1에서는 아래 정도면 충분하다.

- 틀린 문제 리스트 보기
- 틀린 이유 보기
- 번역 다시 보기
- 필요하면 TTS 다시 듣기

즉, 첫 단계는 "오답 다시 확인" 중심이다.

## 9. 보조 기능 UX

## 9.1 번역 UX

번역은 항상 숨김 상태로 시작한다.

학생이 버튼을 눌렀을 때만 열린다.

동작 순서:

1. 로컬 번역이 있으면 바로 표시
2. 없고 API 토글이 OFF면 번역 없음 표시
3. 없고 API 토글이 ON이면 API 번역 요청 가능

## 9.2 TTS UX

TTS는 자동 실행이 아니라 버튼 실행이어야 한다.

권장 버튼:

- `지문 듣기`
- `문항 듣기`
- `보기 듣기`
- `정지`

## 9.3 오답 해석 UX

오답 해석은 결과 화면과 복습 화면에서만 연다.

권장 버튼:

- `짧게 보기`
- `자세히 보기`

이 두 버튼을 나누는 이유:

- 비용을 조절하기 쉽다
- 학생이 필요한 깊이만 선택할 수 있다

## 10. 오류와 빈 상태 UX

학생이 막히지 않게 아래 상태를 분명히 보여줘야 한다.

### 문제 없음

- `현재 선택한 난이도의 문제가 없습니다.`

### 번역 없음

- `이 항목은 아직 번역이 준비되지 않았습니다.`

### API 토글 OFF

- `추가 번역/해설은 API 사용을 켜야 합니다.`

### 저장 실패

- `결과 저장에 실패했습니다. 다시 시도해주세요.`

## 11. v1에서 일부러 단순하게 두는 것

- 로그인 없음
- 프로필 없음
- 기록 대시보드 없음
- 문제 중간 저장 없음
- 학습 통계 화면 없음
- AI가 자동으로 먼저 말을 거는 UX 없음

즉, v1은 "문제 풀기와 오답 확인"에만 집중한다.

## 12. 화면별 핵심 행동 정리

### 시작 화면

- 난이도 고르기
- API 설정 고르기
- 시작하기

### 문제 화면

- 지문 읽기
- 답 고르기
- 필요하면 번역/TTS 쓰기
- 제출하기

### 결과 화면

- 점수 확인
- 틀린 문제 확인
- 오답 해석 보기
- 복습으로 이동

### 복습 화면

- 틀린 문제 다시 보기
- 해석 다시 보기
- 홈으로 돌아가기

## 13. 문서 결론

v1 UX는 아래 흐름으로 고정한다.

```text
시작 -> 풀이 -> 결과 -> 복습
```

그리고 번역, TTS, 오답 해석은 모두 선택 기능으로 둔다.

이 문서를 기준으로 다음 단계에서 화면 명세서와 와이어프레임을 만들 수 있다.
